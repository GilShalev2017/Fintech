1️⃣ Node.js Concurrency (IMPORTANT)

Your question is excellent — let’s clarify:

❗ Node is:

  • Single-threaded (JavaScript execution)
  • Multi-request (event loop + async I/O)

🧠 What actually happens

Scenario:

// Request A
await slowDbCall();

// Request B
await fastDbCall();

Timeline:

Request A starts
↓
await → yields control to event loop

Request B starts while A is waiting
↓
B finishes FIRST
↓
A resumes later

💥 Key insight

👉 Requests can interleave

  • A starts first
  • B finishes first
  • A continues later

❓ Who “wins”?

👉 Depends on timing, not order

Example bug:

task.completed = false; // Request A
task.completed = true;  // Request B

If A resumes AFTER B →
👉 A overwrites B → WRONG result

🔥 This is called:

Race condition

🎯 Senior answer

“Node is single-threaded but async operations allow interleaving between requests, so race conditions can still occur when shared state is mutated.”

2️⃣ Load Balancer & Stateless — Only Docker?

👉 ❌ No — not only Docker

✅ Stateless services mean:

  • No in-memory shared state
  • No reliance on instance memory

Why?

Because you may have:

User → Load Balancer → Server A
                     → Server B

If state is in memory:

  • A doesn’t know B’s data

✅ Load balancer is relevant when:

  • Multiple instances (Docker, VMs, Kubernetes, etc.)
  • Scaling horizontally

🎯 Answer

“Stateless services are required for horizontal scaling, regardless of whether we use Docker, VMs, or cloud instances.”

export const getTasksPrisma = async (): Promise<Task[]> => {
  return prisma.task.findMany();
};

export const createTaskPrisma = async (
  title: string
): Promise<Task> => {
  return prisma.task.create({
    data: { title },
  });
};

export const updateTaskPrisma = async (
  id: number,
  completed: boolean
): Promise<Task | null> => {
  return prisma.task.update({
    where: { id },
    data: { completed },
  });
};

export const deleteTaskPrisma = async (
  id: number
): Promise<void> => {
  await prisma.task.delete({
    where: { id },
  });
};

🎤 Q1 — Concurrent Updates (Race Conditions)

❓ Scenario

Two clients update the same task at the same time.

✅ Strong Answer

“I would prevent inconsistent state using optimistic locking or database transactions, depending on the use case.”

🔍 Expand (what they want to hear)

✅ Option 1 — Optimistic Locking (most common)

Add a version field:

version: number

Update like:

UPDATE tasks
SET completed = true, version = version + 1
WHERE id = 1 AND version = 3

👉 If no rows updated → conflict detected

✅ Option 2 — Transactions (strong consistency)

  • Wrap read + update in a transaction
  • DB ensures atomicity

🧠 Bonus (very strong signal)

🎤 Q2 — If Tasks Were Financial Transactions

✅ Strong Answer

🔥 Key Concepts

1. ❗ Immutability (VERY IMPORTANT)

Instead of:

👉 Do:

transactions.append(...)

✔ Never update — only append records

2. ✅ Idempotency

Prevent duplicate operations:

POST /transactions

👉 Same request won’t execute twice

3. ✅ Strong Consistency

  • Use DB transactions
  • Avoid eventual consistency for money

4. ✅ Audit Trail

  • Full history of changes
  • Who did what and when

5. ✅ Validation & invariants

  • No negative balances
  • No double spending

💥 Strong closing line

“In financial systems, correctness is more important than performance, so I’d favor consistency, immutability, and traceability.”

🔥 Core Question — How Do We Guarantee Correctness in Financial Systems?

This is the most important one.

✅ Strong Structured Answer

“Correctness is achieved through a combination of idempotency, immutability, transactions, and strong validation.”

Same request = same result

✔ Prevent double charges
✔ Use idempotency keys

2️⃣ Transactions (Atomicity)

All-or-nothing:

Debit A → Credit B

3️⃣ Immutability

✔ Never update balances directly
✔ Use ledger entries

4️⃣ Concurrency Control

  • Optimistic locking
  • Row-level locks

5️⃣ Validation (Invariants)

✔ Balance ≥ 0
✔ Currency consistency
✔ Account exists

6️⃣ Observability & Auditing

✔ Logs
✔ Metrics
✔ Traceability
